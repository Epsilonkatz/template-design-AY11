P01 — Page Connexion

Conformité : 79,3%NC : 6Bloquantes : 5

Images

Critère 1.1 A Chaque image porteuse d’information a-t-elle une alternative textuelle ? Bloquante

Constat

L'image du logo AlpiConnect (/images/login/logo_alpiconnect.png) est porteuse d'information — elle identifie le service auquel l'utilisateur se connecte. Elle possède un attribut alt="" qui la désigne comme décorative, ce qui est incorrect. Un utilisateur de lecteur d'écran ne reçoit aucune information sur le service affiché.

Propositions de correction

Ajouter une alternative textuelle pertinente décrivant le logo et le service : <img src="/images/login/logo_alpiconnect.png" alt="AlpiConnect">ou, si le nom du service est déjà présent dans le contexte immédiat (titre h1 ou texte adjacent), l'image peut rester décorative avec alt="" à condition que l'information soit redondante — à vérifier selon le contenu final de la page.

Couleurs

Critère 3.3 AA Dans chaque page web, les couleurs utilisées dans les composants d’interface ou les éléments graphiques porteurs d’informations sont-elles suffisamment contrastées (hors cas particuliers) ? Bloquante

Constat

Le contraste de l'indicateur de focus est insuffisant dans les deux modes. En mode clair (fond blanc #FFFFFF), la couleur de focus #FFBB80 produit un ratio de contraste d'environ 1,79:1, très en dessous du minimum requis de 3:1 pour les composants d'interface (WCAG 1.4.11). Le même problème est constaté en mode sombre. L'information de focus est donc transmise par une couleur dont le contraste est insuffisant par rapport à son arrière-plan.

Présentation de l'information

Critère 10.3 A Dans chaque page web, l’information reste-t-elle compréhensible lorsque les feuilles de styles sont désactivées ? Bloquante

Constat

Sans feuille de style, le logo AlpiConnect (/images/login/logo_alpiconnect.png) apparaît après le bouton "Se connecter" dans l'ordre de lecture du DOM. L'identité visuelle du service est présentée après l'action principale, ce qui crée un ordre de lecture incohérent pour les utilisateurs naviguant sans CSS.

Propositions de correction

Réordonner les éléments dans le DOM pour que le logo précède le bouton "Se connecter". Le CSS peut ensuite gérer l'affichage visuel indépendamment de l'ordre du code source. Dans le HTML, déplacer le bloc <div class="login-card-logo">avant le bloc <div class="btn-connect-wrapper">.

Commentaires

L'ordre des éléments dans le DOM doit être défini et validé dès la phase de conception du design system, indépendamment du rendu CSS. L'ordre de lecture sans CSS doit refléter la hiérarchie logique de l'information pour tous les utilisateurs, notamment ceux utilisant un lecteur d'écran ou naviguant au clavier.

Critère 10.7 A Dans chaque page web, pour chaque élément recevant le focus, la prise de focus est-elle visible ? Bloquante

Constat

L'indicateur de focus n'est pas suffisamment visible dans les deux modes. En mode clair, la couleur #FFBB80 sur fond blanc (#FFFFFF) produit un ratio de contraste d'environ 1,79:1, insuffisant pour permettre à un utilisateur naviguant au clavier de repérer l'élément focalisé. Le même défaut est constaté en mode sombre. La prise de focus n'est donc pas perceptible de manière fiable pour l'ensemble des éléments interactifs de la page.

Propositions de correction

Choisir une couleur de focus offrant un ratio d'au moins 3:1 sur fond blanc en mode clair, et au moins 3:1 sur le fond correspondant en mode sombre. Couleur retenue : #C05C00 (ratio ~4,6:1 sur blanc #FFFFFF — marge suffisante au-dessus du seuil). Implémenter via CSS custom properties pour couvrir les deux modes : :root { --focus-color: #C05C00; } [data-theme="dark"] { --focus-color: #FFD580; } *:focus-visible { outline: 3px solid var(--focus-color); outline-offset: 2px; }

Navigation

Critère 12.1 AA Chaque ensemble de pages dispose-t-il de deux systèmes de navigation différents, au moins (hors cas particuliers) ? Mineure
Critère 12.6 A Les zones de regroupement de contenus présentes dans plusieurs pages web (zones d’en-tête, de navigation principale, de contenu principal, de pied de page et de moteur de recherche) peuvent-elles être atteintes ou évitées ? Bloquante

Constat

Deux non-conformités sur ce critère :

  1. Le lien d'évitement "Aller à la navigation" pointe vers l'ancre #sidebar qui n'existe pas dans le DOM.
  2. Aucune zone <nav> ou role="navigation" n'est définie dans le template, rendant impossible l'atteinte programmatique d'une zone de navigation.

Propositions de correction

  1. Supprimer le lien d'évitement #sidebar ou créer l'élément cible correspondant dans le DOM avec id="sidebar".
  2. Envelopper la navigation principale dans un élément <nav aria-label="Navigation principale"> ou ajouter role="navigation" avec un aria-label pertinent sur le conteneur concerné.

Commentaires

Ce template étant une page de connexion, l'absence de navigation principale est attendue. Pour les pages internes (post-connexion), s'assurer que chaque zone (<header>, <nav>, <main>, <footer>) est correctement balisée et que les skip-links pointent vers des ancres existantes.

Conformité : 79,4%NC : 7Bloquantes : 5Moyennes : 1

Liens

Critère 6.1 A Chaque lien est-il explicite (hors cas particuliers) ? Bloquante

Constat

Le lien pointant vers https://www.alpi40.fr/accessibilite porte un aria-label "RGPD (nouvelle fenêtre)" qui ne correspond pas à sa destination réelle. La page cible traite de l'accessibilité, pas du RGPD. Cet intitulé trompeur ne permet pas à l'utilisateur de comprendre la destination du lien.

Propositions de correction

Corriger l'aria-label pour refléter la destination réelle : aria-label="Accessibilité (nouvelle fenêtre)". Si la page cible traite effectivement du RGPD, corriger l'URL en conséquence.

Commentaires

Le lien "Déconnexion" (href="/login") est conforme RGAA — intitulé explicite, 6.1 et 6.2 respectés. Cependant, si la déconnexion est gérée par une action POST avec token CSRF (cas standard sous Symfony), un <a> est sémantiquement incorrect et potentiellement non sécurisé. Recommandation : confirmer le comportement serveur avec les développeurs. Si la déconnexion est bien une action POST, remplacer par : <form action="/logout" method="post"><input type="hidden" name="_csrf_token" value="{{ csrf_token('logout') }}"><button type="submit" class="btn btn-primary">Déconnexion</button></form>

Scripts

Critère 7.1 A Chaque script est-il, si nécessaire, compatible avec les technologies d’assistance ? Bloquante

Constat

Le bouton de changement de thème (id="themeToggle") ne restitue pas son état courant aux technologies d'assistance. L'aria-label est statique ("Changer le thème") : l'utilisateur n'entend jamais quel thème est actuellement actif, ni lors de la prise de focus ni après activation. L'attribut aria-pressed est absent, privant le lecteur d'écran de l'information d'état (activé/non activé). L'icône Phosphor (<i>) ne porte pas aria-hidden="true", ce qui peut entraîner une vocalisation parasite du contenu généré par la police d'icônes.

Propositions de correction

  1. Ajouter aria-pressed et un aria-label dynamique reflétant l'action à venir selon le thème courant.
  2. Ajouter aria-hidden="true" sur l'icône. HTML initial : <button class="btn-theme-toggle" id="themeToggle" aria-label="Activer le thème sombre" aria-pressed="false"><i class="ph ph-sun" id="themeIcon" aria-hidden="true"></i></button> JS — modifier la fonction applyTheme() : function applyTheme(theme) { html.setAttribute("data-theme", theme); const isDark = theme === "dark"; icon.className = isDark ? "ph ph-moon" : "ph ph-sun"; themeBtn.setAttribute("aria-pressed", isDark ? "true" : "false"); themeBtn.setAttribute("aria-label", isDark ? "Activer le thème clair" : "Activer le thème sombre"); currentTheme = theme; } Résultat : au chargement en thème clair, le lecteur d'écran annonce "Activer le thème sombre, bouton non activé". Après bascule, il annonce "Activer le thème clair, bouton activé".
Critère 7.3 A Chaque script est-il contrôlable par le clavier et par tout dispositif de pointage (hors cas particuliers) ? Bloquante

Constat

À l'activation du bouton burger, le menu latéral s'ouvre mais le focus reste sur le bouton burger. L'utilisateur naviguant au clavier ou avec un lecteur d'écran n'est pas repositionné sur le premier élément de la navigation, rendant l'accès au contenu du menu fastidieux.

Propositions de correction

Déplacer programmatiquement le focus vers le premier élément interactif de la sidebar à l'ouverture du menu, et le renvoyer sur le bouton burger à la fermeture. Dans la fonction toggleSidebar(), ajouter après l'ouverture :

Commentaires

Exemple de code
if (isOpen) {
  const firstLink = sidebar.querySelector('a, button');
  if (firstLink) firstLink.focus();
} else {
  btn.focus();
}

Présentation de l'information

Critère 10.2 A Dans chaque page web, le contenu visible porteur d’information reste-t-il présent lorsque les feuilles de styles sont désactivées ? Moyenne

Constat

Sans feuille de style, le bouton burger (<button id="burgerBtn">) apparaît visuellement vide. Il ne contient qu'un SVG inline qui ne s'affiche plus correctement sans CSS. Un utilisateur voyant naviguant sans CSS ne peut pas identifier la fonction de ce bouton. Le nom accessible aria-label="Ouvrir le menu" est bien restitué aux technologies d'assistance mais n'est pas visible dans le contenu de la page.

Propositions de correction

Ajouter un texte masqué visuellement dans le DOM via une classe sr-only (visually-hidden), visible uniquement sans CSS : <button

Exemple de code
class="burger-btn"
  onclick="toggleSidebar()"
  aria-label="Ouvrir le menu"
  aria-expanded="false"
  id="burgerBtn"
>
  <svg ...></svg>
  <span class="sr-only">Ouvrir le menu</span>
</button>

.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

Commentaires

Le sr-only masque le texte visuellement avec CSS — donc avec CSS activé, l'utilisateur ne le voit pas. Sans CSS, la règle sr-only ne s'applique plus et le texte "Ouvrir le menu" devient visible dans le bouton.

Consultation

Critère 13.11 A Dans chaque page web, les actions déclenchées au moyen d’un dispositif de pointage sur un point unique de l’écran peuvent-elles faire l’objet d’une annulation (hors cas particuliers) ? Bloquante

Constat

Le <div id="sidebarOverlay"> déclenche la fermeture du menu via onclick="closeSidebar()". Cette action sur pointage en point unique ne dispose pas de mécanisme d'annulation — il n'est pas possible de glisser hors de la cible pour annuler l'action avant de relâcher le pointeur.

Propositions de correction

Remplacer le gestionnaire onclick par un gestionnaire sur l'événement pointerup avec vérification que le pointeur n'a pas quitté la cible, ou utiliser un <button> natif qui gère nativement l'annulation par glissement hors de la cible. Exemple : remplacer onclick="closeSidebar()" par un écouteur pointerup vérifiant event.target === overlay avant d'exécuter la fermeture.

Implémenter via CSS custom properties pour couvrir les deux modes : :root { --focus-color: #C05C00; } [data-theme="dark"] { --focus-color: #FFD580; } *:focus-visible { outline: 3px solid var(--focus-color); outline-offset: 2px; } L'activation du skip-link #sidebar, déclencher programmatiquement l'ouverture de la sidebar si elle est masquée. Exemple : dans le gestionnaire du skip-link, appeler toggleSidebar() si la sidebar ne possède pas la classe open au moment de la navigation vers l'ancre

Thématique 11 — Formulaires

Champs obligatoires et gestion des erreurs

Champs obligatoires

Les champs obligatoires doivent être indiqués de manière nominative ou disposer de l'un des attributs suivants :

  • aria-required="true"
  • required

Indication visuelle

Si les champs obligatoires ne sont pas indiqués nominativement, il faut prévoir une indication visuelle explicite dans l'étiquette du champ ou un texte associé via aria-describedby.

Un astérisque seul sans définition n'est pas suffisant. Il faut définir l'astérisque avant le premier champ obligatoire :

Exemple conforme
<p>* champ obligatoire</p>

Messages d'erreur et gestion du focus

Rédiger un message d'erreur utile

Un bon message d'erreur dit à l'utilisateur ce qui ne va pas et comment corriger. Il ne répète pas le label du champ et n'utilise pas de jargon technique.

  • Champ vide : « Le prénom est obligatoire. » — pas « Valeur manquante. »
  • Format incorrect : « Saisissez une adresse email valide, par exemple nom@domaine.fr. » — pas « Format invalide. »
  • Longueur insuffisante : « Le message doit contenir au moins 10 caractères. » — pas « Trop court. »

Structure HTML d'un champ avec erreur

Avant soumission (état normal)
<label for="email">Courriel <span aria-hidden="true">*</span></label>
<input id="email" type="email" autocomplete="email"
       required aria-required="true">
<p id="email-error" role="alert" class="error-msg" hidden></p>
Après soumission (état erreur)
<label for="email">Courriel <span aria-hidden="true">*</span></label>
<input id="email" type="email" autocomplete="email"
       required aria-required="true"
       aria-invalid="true"
       aria-describedby="email-error">
<p id="email-error" role="alert" class="error-msg">
  Saisissez une adresse email valide, par exemple nom@domaine.fr.
</p>

JavaScript — validation à la soumission avec gestion du focus

Après soumission avec erreurs, le focus est déplacé sur le premier champ en erreur — pas de résumé global.

Exemple complet
const validationRules = {
  firstName: {
    validate: val => val.trim().length > 0,
    message:  "Le prénom est obligatoire."
  },
  email: {
    validate: val => /^[^@]+@[^@]+\.[^@]+$/.test(val),
    message:  "Saisissez une adresse email valide, par exemple nom@domaine.fr."
  },
  message: {
    validate: val => val.trim().length >= 10,
    message:  "Le message doit contenir au moins 10 caractères."
  }
};

function setFieldError(fieldId, message) {
  const field   = document.getElementById(fieldId);
  const errorEl = document.getElementById(fieldId + "-error");
  field.setAttribute("aria-invalid", "true");
  field.setAttribute("aria-describedby", fieldId + "-error");
  errorEl.textContent = message;
  errorEl.removeAttribute("hidden");
}

function clearFieldError(fieldId) {
  const field   = document.getElementById(fieldId);
  const errorEl = document.getElementById(fieldId + "-error");
  field.removeAttribute("aria-invalid");
  field.removeAttribute("aria-describedby");
  errorEl.textContent = "";
  errorEl.setAttribute("hidden", "");
}

function validateForm(e) {
  e.preventDefault();
  let firstErrorId = null;

  for (const [fieldId, rule] of Object.entries(validationRules)) {
    const field = document.getElementById(fieldId);
    if (!rule.validate(field.value)) {
      setFieldError(fieldId, rule.message);
      if (!firstErrorId) firstErrorId = fieldId;
    } else {
      clearFieldError(fieldId);
    }
  }

  if (firstErrorId) {
    document.getElementById(firstErrorId).focus();
  } else {
    e.target.submit();
  }
}

document.querySelector("form").addEventListener("submit", validateForm);

Les attributs ARIA

role="status"

Indique que la zone contient un message de statut non urgent (confirmations, mises à jour). Informe les technologies d'assistance de la nature informationnelle de la zone. Équivalent implicite de aria-live="polite".

Exemples d'usage : « Enregistré », « 2 produits dans le panier », « Slide 2 sur 3 ».

aria-live="polite"

Indique aux lecteurs d'écran d'annoncer les changements dans une zone dynamique. La valeur polite signifie que l'annonce attend la fin de la lecture en cours, sans interrompre l'utilisateur. Utiliser aria-live="assertive" uniquement pour les urgences.

aria-expanded

Indique l'état ouvert (true) ou fermé (false) d'un composant pliable (accordéon, menu déroulant). Doit être mis à jour dynamiquement via JavaScript à chaque interaction.

aria-controls

Référence l'identifiant (id) du contenu contrôlé par un bouton. Permet aux technologies d'assistance de naviguer directement vers le contenu associé.

class="sr-only"

Convention CSS permettant de masquer visuellement un élément tout en le conservant dans le flux d'accessibilité — les lecteurs d'écran le lisent, les utilisateurs voyants ne le voient pas.

À ne pas confondre avec display: none ou visibility: hidden qui masquent l'élément à la fois visuellement et aux technologies d'assistance.

Quand l'utiliser

  • Compléter un intitulé visuel insuffisant — ex. un bouton avec une icône seule : <button><svg .../><span class="sr-only">Fermer</span></button>
  • Contextualiser un lien ambigu — ex. plusieurs liens « Lire la suite » sur une même page : <a href="...">Lire la suite <span class="sr-only">de l'article : Nouveau guide numérique</span></a>
  • Indiquer un champ obligatoire sans astérisque visible seul : <label>Prénom <span class="sr-only">(obligatoire)</span></label>
  • Titrer une région sans titre visible — ex. une <section> dont le titre est purement graphique : <h2 class="sr-only">Chiffres clés</h2>
  • Annoncer un changement de contexte — ex. avertir qu'un lien ouvre un nouvel onglet quand l'information ne figure pas dans l'intitulé visible.

Quand ne pas l'utiliser

  • Pour masquer du contenu qui n'a pas de raison d'être restitué aux AT — utiliser aria-hidden="true" à la place.
  • Pour remplacer un vrai label visible — si un champ mérite une étiquette, celle-ci doit être visible pour tous les utilisateurs.
Implémentation CSS
.sr-only {
  position: absolute;
  width: 1px; height: 1px;
  padding: 0; margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

aria-current="page"

À utiliser dans les éléments de navigation pour indiquer la page active. Annoncé par les lecteurs d'écran pour situer l'utilisateur dans la structure de navigation.
